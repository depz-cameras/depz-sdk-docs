Cross-language SDK by example — the same morphisms in every language

Companion to `CROSS_LANGUAGE_CATEGORY.md` (the abstract theory). This one is concrete: the same SDK program written in every binding, side by side, plus the morphism→method table that makes the functor visible. If the theory doc says "each binding is a functor `F_L : Depz → Lang`", this doc shows F_L applied to one program.

Status (v2.3): the flat C ABI depz_c_api.h has 69 generating morphisms. Coverage: Java FFM 69/69, ctypes-Python 69/69, Node N-API 69/69 (the 9 discovery/cache helpers are bound natively in the addon too — the binding is self-sufficient). nanobind-Python binds the C++ classes directly (a richer, separate presentation).

2. The same program in every language

One task — build the depth pipeline, start it, pull one frame, read the disparity, stop — stripped to the SDK spine (no rendering).

3. The functor, made into a table

Each row is one morphism of Depz; each column is F_L applied to it. Reading across a row is exactly the cross-language isomorphism φ = F_b ∘ F_a⁻¹.

The only thing that varies across a row is the name, and the name is a pure function of the language convention — never of the morphism:

Convention C ABI C++ / Java / Node ctypes / nanobind Python
word style snake_case + depz_ prefix camelCase snake_case
set_filter_param depz_pipeline_set_filter_param_b setFilterParam set_filter_param
seek_us depz_pipeline_seek seekUs seek_us

Renaming is a relabeling of objects/arrows — an isomorphism of the underlying directed graph. It does not touch composition, so it carries the lifecycle laws (create ; destroy = id, start ⟶ get, image-only-on-live- frame) across unchanged.

4. Homogeneity, categorically

For program p (any valid arrow-composite in Depz) and bindings F_a, F_b:

Depz ∋ p
│
F_a(p) │ F_b(p) F_a, F_b are full + faithful + total
▼ │ ▼ ⟹ each is an iso onto its image
image(F_a) ≅ image(F_b) φ = F_b ∘ F_a⁻¹ is the language↔language iso
│
observe ∘ ⟦·⟧ must AGREE for every p

Two regimes for the agreement — and we measured both:

disparity (min / max / mean) φ preserves observables?
ctypes-Python → 7.5795 / 38.0242 / 19.3254 ┐ SAME functor (same 69 C fns)
Java (C ABI) → 7.5795 / 38.0242 / 19.3254 ┘ DEFINITIONAL iso, Δ = 0.0000
Node (C ABI) → 7.5795 / 38.0242 / 19.3254 ┘ (same factorisation)
nanobind-Python → 7.6842 / 36.8214 / 19.4459 ┐ DIFFERENT functor (C++ classes)
Java (C ABI) → 7.5795 / 38.0242 / 19.3254 ┘ NATURAL iso, Δ within ML drift
  • Definitional iso (C · C++-via-ABI · Java · ctypes-Py · Node): all factor through the one set of 69 C functions, so φ is an equality of observables, byte-for-byte. Proven once on the generators ⟹ holds on every composite by functoriality (we don't test infinitely many programs — we test the generators + the laws).
  • Natural iso (nanobind-Python, C++ core): a different presentation of the same ⟦·⟧ (the C++ classes, not the C ABI). φ preserves observables only up to sub-pixel ML drift — equal as SDK behaviour, not by definition.

Rendering (OpenGL cloud, ImageIO, Swing, the browser MJPEG of depz.start_stream) is a functor into Display_lang that we never require to commute — it is outside Depz.
