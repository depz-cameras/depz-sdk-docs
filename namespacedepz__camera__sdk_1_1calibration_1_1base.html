Detailed Description

Calibration ABC + registry + cereal-shape persistence + canvas math.

Concrete impls live in sibling modules (`opencv_basic`, `opencv_fisheye`)
and register themselves with a stable ``KIND`` string. Persistence:
``to_dict`` / ``from_dict`` round-trip through the registry — recordings
are self-describing and online recalibrators can ship state updates as
opaque dicts.

All concrete impls share a single wire format: cereal-shaped
``{<sdk_params>, "cameras": [left, right]}`` — no outer wrap. The
calibration class is identified on read by the ``polymorphic_name``
field on each camera's lens model — no ``kind`` tag, no structural
sniffing. ``DepzCameraInfo`` keeps the calibration under
``"base_calibration"`` in its own envelope; the server's device-info
endpoint emits the same shape.
Each subclass declares:

  - :attr:`POLYMORPHIC_NAME` — the lens-model string it owns
    (e.g. ``"LENSMODEL_OPENCV8"``, ``"LENSMODEL_OPENCV_FISHEYE"``).
  - :meth:`controls` — list of :class:`~depz_camera_sdk._controls.Control`
    descriptors for runtime params (round-trip through ``to_dict`` /
    ``from_dict`` alongside ``cameras``). Same self-describing shape
    as ``Filter.parameters()`` and ``DepthInferenceBase.controls()``.

Subclass hooks (small, focused):
  - :meth:`_extract_distortion`   — read distortion coeffs from a camera entry.
  - :meth:`_distortion_params_dict` — format coeffs back into the params dict.
  - :meth:`_compute_rectification` — class-specific rectification math.
  - :meth:`_apply_partial`         — class-specific partial-update fields.
  - :meth:`controls`               — class-specific runtime params.
  - :meth:`control_set`               — only when special semantics needed
    (sentinels, cross-control constraints); default works via
    :class:`~depz_camera_sdk._controls.ControlOwner` + base override that
    triggers :meth:`_compute_rectification`.
  - :meth:`adapt_to_resolution`    — class-specific size adaptation.

Everything else — cereal extraction, role/index splitting, K/R/T parsing,
cereal emit, dispatcher, get/set/coerce/to_dict — is on the base and
works identically for every concrete class.

◆ register_calibration()

type depz_camera_sdk.calibration.base.register_calibration ( type  cls) 
Decorator: register a concrete ``Calibration`` subclass.

The class must declare its own ``POLYMORPHIC_NAME`` (the cereal
lens-model identifier such as ``"LENSMODEL_OPENCV8"``). The registry
is keyed by that name — the same string that appears on
``cameras[i].model.polymorphic_name`` in the wire format and the
sole signal :meth:`Calibration.from_dict` uses to dispatch.

Idempotent re-registration of the *same class object* is a no-op
(supports notebook hot-reload where the module is re-imported and
the decorator runs again on the same class). Registering a
*different* class under an already-claimed name raises — that's
almost always a collision the operator wants to know about.
