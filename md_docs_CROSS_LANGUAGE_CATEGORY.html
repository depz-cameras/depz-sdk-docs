Cross-language SDK — the category, the functors, the isomorphism

How the one C++ core is presented to every language, and why the bindings are guaranteed isomorphic as SDK usage — while presentation (OpenGL, ImageIO, Swing) is deliberately left free.

TL;DR. depz_c_api.h is the generating signature of a category Depz. Each binding is a full, faithful, total functor Depz → Lang, so each is an isomorphism onto its image. Composing two of them gives an isomorphism between language surfaces; the conformance test checks that isomorphism preserves observable values. Presentation lives in a separate category we never require to commute.

2. The category Depz

  • Objects — the opaque handle types plus the value/record types (Image, Metrics, Knob, CanvasLayout, CameraInfo, …).
  • Morphisms — the operations of depz_c_api.h, typed by their C signature, e.g.
    • pipeline_create : Source ⊗ Calibration ⊗ Inference ⊗ Config → Pipeline
    • get : Pipeline → Frame
    • image : Frame ⊗ Stream → Image
    • sample_depth : Frame ⊗ ℕ² → ℝ
  • Composition — valid operation sequences, subject to the lifecycle laws: create ; destroy = id (no leak/double-free), start precedes get, a Frame is read before destroy, replay ops are no-ops on a live source, etc. (Handles are linear resources → Depz is a symmetric monoidal category with explicit create/discard.)

depz_c_api.h is the finite presentation: its 69 functions are the generators; the lifecycle laws are the relations.

3. Each binding is a functor — and which kind

A binding F_L : Depz → Lang maps each object to a wrapper type and each morphism to a method. The properties we want and how they're guaranteed:

Functor property What it means for the binding Guaranteed by
total on generators every C-ABI op has a method check_abi.py coverage = 69/69 = 100 %
faithful (injective on morphisms) distinct ops → distinct methods; arity preserved check_abi.py arity match + 1 handle : 1 C function
full (surjective on hom-sets) no method outside the SDK morphisms; sugar is derived from generators design rule: wrappers only compose C-ABI calls
bijective on objects each handle type ↔ exactly one wrapper class design (Pipeline↔DepzPipeline, …)

Full + faithful + bijective-on-objects ⟹ **F_L is an isomorphism of categories onto its image**: Depz ≅ image(F_L) ⊂ Lang.

Today: **F_java is full, faithful and total (100 %)** — verified. So the Java surface is isomorphic to Depz itself.

4. The isomorphism <em>between languages</em>

Two bindings give two isomorphisms onto their images:

image(F_java) ≅ Depz ≅ image(F_py)
╰──────────────────────╯
the language surfaces are isomorphic categories;
the iso is φ = F_py ∘ F_java⁻¹ .

This is a structural (syntactic) isomorphism. The remaining question is semantic: does φ preserve observable values? That is exactly a natural isomorphism η : observe∘⟦·⟧∘F_java ⇒ observe∘⟦·⟧∘F_py on the generating programs. Functoriality then extends it to all programs:

If two functors agree on the generators and preserve composition, they agree on every composite. A finite conformance suite over the generators

  • the lifecycle laws ⟹ equivalence on the infinitely many programs.

That is why we don't (and can't) test every program — we test the generators and the laws.

6. The one honest caveat — Python

F_java factors through depz_c_api.h (it literally calls those 65 functions). The production F_python factors through nanobind over the C++ classes — a different presentation of the same ⟦·⟧. So Python and Java are isomorphic only up to natural isomorphism (equal observables within tolerance — the conformance test showed sub-pixel ML drift), not by definition.

To make it definitional (φ an equality, not a tested iso), route a Python binding through the same C ABI (ctypes over depz_c_api.h). Then F_python and F_java are the same functor in two syntaxes, and cross-language equality holds by construction.

This is done and verified — tests/conformance/depz_cabi.py is exactly that ctypes functor, and run.sh compares both Python bindings to Java:

disparity (min / max / mean) verdict
nanobind-Python → 7.6842 / 36.8214 / 19.4459 ┐ different functor
Java (C ABI) → 7.5795 / 38.0242 / 19.3254 ┘ natural iso (Δ within drift)
ctypes-Python → 7.5795 / 38.0242 / 19.3254 ┐ SAME functor
Java (C ABI) → 7.5795 / 38.0242 / 19.3254 ┘ definitional iso (Δ = 0.0000)

The ctypes-Python numbers equal Java's byte-for-byte and differ from nanobind-Python — proving the drift is the functor, not the language. Once Python factors through the one C ABI, equality is by construction.

7. What "100 %" requires, concretely

  1. Total functor — every C-ABI op wrapped. ✅ Java 100 % (check_abi.py).
  2. Observable equality on generators — ✅ definitional for the compute morphism (ctypes-Python ≡ Java, Δ = 0); deterministic parse exact; nanobind-Python within drift. More generators/laws can be added the same way.
  3. Same factorisation — bindings route through the one C ABI. ✅ Java; ✅ demonstrated for Python (depz_cabi.py); production Python stays nanobind (a different-but-naturally-isomorphic functor by design).

Presentation (OpenGL camera/cloud, ImageIO) is a functor into Display_lang that we intentionally leave free — it is outside Depz and never enters the isomorphism.

8. The same structure as GoF design patterns

The category-theoretic picture above is the formal version of a handful of classic design patterns — most visibly Builder and Facade. The patterns are the informal name for exactly the structure the functor makes precise:

GoF pattern Where in the SDK Categorical counterpart
Facade each OOP binding (DepzPipeline / DepthPipeline) hides the 69 flat C functions + the source⊗calib⊗inference⊗pipeline lifecycle behind one object — facade over facade: the C ABI already facades the C++ core the object map of the functor F_L (handle → wrapper)
Builder construction: Java fluent builder().camera().inference().build(); Node build({…}), ctypes/nanobind build(**kw) (the named-args variant) ergonomic wrapper over the morphism create : Source⊗Calib⊗Inference⊗Config → Pipeline
Factory Method static factories DepzSource.camera/mkv/pipeline, DepzCalibration.fromFile/fromJson/fromWeb, DepzInference.create the sub-object constructors that feed create
Adapter the binding layer itself — Panama FFM / N-API / ctypes / nanobind each adapt the one C ABI to a language idiom the functor F_L as a syntax change, not a semantics change
Bridge the C ABI itself — decouples abstraction (language API) from implementation (C++ core): swap the core, all 6 languages unaffected the generating signature depz_c_api.h that everything factors through
Disposable / RAII try-with-resources / with / using / dtor / close() the composition law create ; destroy = id (handles are linear resources)
Strategy in the core: IStereoInference (engines), IDepthFilter (6 filters) pluggable morphisms behind a fixed interface
Iterator (pull) the get() loop draining BlockQueue<FramePacket> repeated application of get : Pipeline → Frame

Two honest notes:

  • Builder vs named-args. Java is the only true fluent Builder; the Python/Node build(named-args) is strictly a parameterised factory, not a step-wise Builder. This is the one construction divergence we deliberately keep — Java idiom prefers builders, Python prefers kwargs (see CROSS_LANGUAGE_DIVERGENCE.md).
  • Facade + Builder dominate because that is precisely a binding's job: hide the flat surface (Facade) and assemble the complex object ergonomically (Builder). Everything else (Factory/Adapter/Bridge/Disposable/Strategy) is the supporting cast — and each maps cleanly onto a piece of the functor.
